# 萨卡兹语大模型微调计划 (Sarkaz LLM Fine-Tuning Plan)

本文档旨在规划如何利用大型语言模型（如 Qwen 小模型）的强大语境能力，解决《明日方舟：终末地》中”萨卡兹语”到中文的解码与翻译问题。

- [x] 语料清洗与平行数据生成（333k 明日方舟句子）
- [x] Tokenizer 训练（SPM + 投影 + 硬编码，词表 168k）
- [x] 0.6B smoke test 验证流程
- [ ] 4B 云端训练（目标）
- [ ] 终末地领域 LoRA（待语料收集）

## 1. 背景与核心挑战

萨卡兹语采用极其简单但信息损耗极大的映射规则：`ord(char) % 56`。
这意味着数以千计的常用中文字符被压缩到了 26 个英文字母中，存在海量的“哈希碰撞”。传统的基于字典或局部概率的模型无法有效应对这种严重的歧义。

**核心解决方案**：引入具备“世界知识”和长程上下文感知能力的 LLM，通过上下文语境进行极端的语义消歧。

## 2. 阶段一：语料整理与平行数据集构建

高质量的语料是微调成功的基础。我们需要构建规模大且富有“终末地”语境的平行语料库。

*   **原始数据源**：
    *   项目现有的 `wordlist.txt`（常用词）、`single_char.txt`（常用字）。
    *   现有的 `wiki2019zh.json`，提供字频统计信息（可用于频率加权采样），但不含可用文本。
    *   提取的《明日方舟》及《明日方舟：终末地》剧情文案、道具描述（高优先级）。
*   **平行语料合成**：
    *   利用核心算法生成 `(中文, 萨卡兹语)` 对。
    *   引入噪声：在生成的部分萨卡兹语中人为增加轻微的拼写错误或缺失，增强模型在 OCR 提取场景下的鲁棒性。
*   **SFT (指令微调) 格式构建**：
    构建类似如下的 Prompt-Response 格式：
    ```json
    {
      "instruction": "请结合《明日方舟：终末地》的世界观，将以下萨卡兹密文翻译为中文原意。",
      "input": "ytqqrvqbsbtjyrernx",
      "output": "你好，管理员，是来提交活动菜品的吗？"
    }
    ```

## 3. 阶段二：Tokenizer 的重构与优化

这是最关键的一步，决定了模型是“真正理解词义”还是“盲目猜测字母组合”。
现有的开源模型 Tokenizer 在遇到 `ytqq` 这样的输入时，倾向于按不成逻辑的英文子词进行切片，这会彻底破坏萨卡兹语与原始中文字符形态的对应关系。

*   **中文明文与萨卡兹密文混合联合训练**：
    *   **不能仅用萨卡兹语训练 Tokenizer**。必须将**原始中文文本**与**对应的萨卡兹语文本**混合，作为语料喂给 BPE (Byte-Pair Encoding) 或 WordPiece 训练器。
    *   利用原始中文语料中丰富的词汇边界统计规律，强制 Tokenizer 学习到“管理员”及其映射 `sbt` 应该作为一个完整的 Token 存在。
*   **领域字典注入**：
    *   读取 `LLMTools/MCPServer/endfield_words.txt` 及其对应的萨卡兹编码，将其作为强制保留的特殊 Token (User-defined symbols) 注入到词表中，确保关键名词不会被切碎。

## 4. 阶段三：模型微调策略 (Fine-Tuning)

*   **基座模型选择**：选择 Qwen 等对中文和多语言支持极好，且具备小参数量版本（如 0.5B, 1.8B 等）的模型，以便在消费级硬件上进行研究。

采用 **两阶段微调架构 (Two-Stage Curriculum Learning)**，将语言的”通用映射规律”与”领域特定语境”解耦（注：此处”两阶段微调”指微调本身的两个步骤，与前文语料整理、Tokenizer 改造等阶段共同构成完整的五阶段流程）：

*   **第一步：基础语言能力注入 (General Base-Tuning)**
    *   **语料**：使用海量通用中文语料（新闻、日常对话等）与其对应的萨卡兹语对。
    *   **方法**：在此阶段可采用较大秩 (Rank) 的 LoRA，或者在使用新 Tokenizer 的情况下解冻 Embedding 层 + LoRA。
    *   **目的**：让模型首先掌握 `ord(char) % 56` 这一底层加密规则的统计学规律，学会像普通中国人一样区分常见的同音/同构词错觉（如 `的/地/得`，`你/我/他`的推断）。
    *   **Go/No-Go 标准**：在通用验证集上 CER < 30% 即视为路径可行，可进入第二步；否则需调整数据规模或超参数。
*   **第二步：领域知识注入 (Domain Adaptation via LoRA)**
    *   **语料**：提取《明日方舟》及《明日方舟：终末地》的纯正剧情文案、世界观设定、角色对话构建成对语料。
    *   **方法**：在第一步训练好的权重基础上，专门**外挂（且可热插拔）一个针对“终末地”领域的 LoRA 模块**。
    *   **目的**：利用较小的算力快速将模型的语境和画风“掰”向终末地世界观（例如，在所有称呼候选词中优先选择“管理员”、“干员”）。
    *   **优势**：极度灵活。未来如果需要将这套”看字母猜中文”的引擎用于其他游戏或文本群，只需卸载终末地的 LoRA，换上新领域的 LoRA 即可。
    *   **Go/No-Go 标准**：在领域验证集上 CER < 15%，专有名词命中率 > 95%。

*   **思维链 (Chain of Thought, CoT) 尝试**：
    为了激发极限的语境推理，可以构建部分引导性数据，让模型在给出最终结果前，先输出思考过程：
    *   *模型输出示例*：`[思考] 输入包含片段 sbt，在终末地语境中可能是“管理员”。结合开头 ytqq (你好)，这是日常对话问候。 [翻译] 你好，管理员...`

## 5. 阶段四：动态知识注入与外部挂载 (RAG 架构)

你提到的一个关键盲区是：**游戏会不断更新，引入脱离训练集的新名词和新机制。反复微调模型成本极高且不现实。**
因此，在完全依赖 LLM 微调之外，需要引入 **检索增强生成 (RAG) 和动态外挂词典** 的混合架构。

*   **动态字典树扫描 (Aho-Corasick / Trie)**：
    *   维护一个动态词库（如 `endfield_words.txt` 的增强版）。当有新名词（如新角色“佩丽卡”）出现时，热更新进同文件夹下的纯文本字典即可。
    *   构建一个本地 Python 脚本引擎。在将萨卡兹密文发送给 LLM 之前，先用自动机（Aho-Corasick 或 Trie 树）在密文中扫描出所有可能的“字典命中片段”。
*   **Prompt Hint 注入 (动态提示词)**：
    *   在推理阶段，把本地脚本扫描出的候选词，作为线索（Hints）直接塞进 LLM 的上下文。
    *   *推理期 Prompt 设计示例*：
        ```text
        系统：请结合《明日方舟：终末地》语境翻译以下萨卡兹语。
        输入密文：ytqqrvqbsbtjyrernx
        【实时外部线索】：我们在密文中扫描到了以下可能的专有名词组合，请在翻译时予以参考：
        - sbt -> [管理员, 某新机制A]
        - jy -> [某新角色B]
        
        模型输出：...
        ```
*   **微调与 RAG 的配合**：
    微调赋予了 LLM “翻译这门语言的语法和语感”，而 RAG 赐予了模型“当下的最新名词记忆”。微调只需要偶尔做（更新骨架），而外挂词典可以每分钟都在添加新词。

## 6. 阶段五：评估、验证与部署

*   **评估指标 (Metrics)**：
    *   由于是一对一映射，传统的 BLEU 或 ROUGE 可能不够直观。可以计算**字级别准确率 (Character Error Rate, CER)**。
*   **模型集成**：
    *   微调完成后，将新模型通过 vLLM 或 llama.cpp 在本地部署。
    *   更新现有的 MCP Server（`Sarkaz_tools.py`），加入调用本地微调模型的工具接口，真正实现“AI 与定制化工具协作翻译萨卡兹语”。
